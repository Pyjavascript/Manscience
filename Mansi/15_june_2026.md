# 5. Architecture Review

**Overall architecture:** Layered, provider-agnostic design:

```
main.py (CLI)  /  app/api/* (REST)
        тФВ
        тЦ╝
ChatService.handle_message()   тЖР orchestration
        тФВ
   тФМтФАтФАтФАтФАтФ╝тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР
   тЦ╝    тЦ╝             тЦ╝
Memory  PromptBuilder  LLMClient (ABC)
Service  (pure fn)         тФВ
                       OpenAIClient (concrete)
                            тФВ
                       app.config.settings (env config)
```

- **Major modules:** `config` (settings/validation), `llm` (provider abstraction + OpenAI impl), `services` (memory, prompt building, orchestration), `api` (FastAPI surface), plus empty stubs for `rag`, `rules`, `integrations`.
- **Dependency flow:** Both entry points (`main.py`, `app/api/chat.py`) depend only on `ChatService`. `ChatService` depends on the `LLMClient` interface (not a concrete provider) and on `MemoryService`/`build_messages`. Only `app/llm/openai_client.py` imports the `openai` SDK. Only `app/config/settings.py` reads `os.environ`.
- **Separation of concerns:** Clean тАФ config, memory, prompt construction, and provider I/O are each isolated and independently tested. `prompt_builder.py` is explicitly a pure function with no I/O.
- **Good practices observed:**
  - Single source of truth for config (`settings` singleton, frozen dataclass, masked `__repr__` for secrets).
  - Abstract `LLMClient` interface enables future multi-provider support without touching `ChatService`.
  - Retry/backoff with jitter for transient OpenAI errors; distinct handling for auth vs. bad-request vs. retryable errors.
  - User-safe error messages (`ChatServiceError`) never leak raw provider exception details to callers.
- **Design gaps relative to a "production-ready assistant":**
  - No session/user identity concept anywhere тАФ `MemoryService` is single-session, in-process, non-persistent.
  - FastAPI layer has a single shared `ChatService`/memory instance тАФ a multi-user concern not yet addressed (acknowledged scope limitation per the FastAPI spec, not a bug per se for current phase).

- **TODO/FIXME/XXX/HACK comments:** None found in `app/`, `main.py`, or `tests/` (grep returned no matches).

  - In `app/api/chat.py`, the shared module-level `_chat_service` means conversation memory is global across all API clients тАФ likely fine for a single-user demo, but a correctness issue for any multi-user deployment. Not a bug against current stated scope, but worth flagging for the next phase.
  - `MemoryService.append` raises `ValueError` for empty `content` тАФ if an LLM ever returns an empty string (e.g., a finish_reason like `content_filter`), `ChatService.handle_message` would raise this as an unhandled `Exception` тЖТ caught by the generic `except Exception` in `handle_message` тЖТ surfaced as "An unexpected error occurred." This is *handled*, but the user gets a generic error rather than something actionable. Minor.
- **Missing validation:**
  - `ChatRequest.message` only enforces `min_length=1` тАФ whitespace-only messages pass Pydantic validation and are caught later by `ChatService` (returns 400) тАФ this works correctly per `test_chat_wraps_chat_service_error_as_400`, so not a bug, just worth noting validation happens in two layers.
  - No maximum message length enforced anywhere (could allow very large prompts/cost).
- **Security concerns:**
  - `/chat` and `/health` have no authentication тАФ acceptable for local dev, explicitly flagged as a gap in `docs/specs/api-specification.md` ┬з4.
  - `.env` contains a real OpenAI API key and is correctly excluded via `.gitignore` (`.env`, `.env.*`, with `!.env.example` exception) тАФ good practice, verified.
  - No rate limiting on `/chat`, which calls a billed external API тАФ could allow cost-exhaustion if exposed publicly.
